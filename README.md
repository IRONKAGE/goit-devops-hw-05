# **Завдання 5: IaC. Terraform**

## **Опис завдання:**

Ваше домашнє завдання полягає у створенні Terraform-структури для інфраструктури на AWS у **новій директорії** `lesson-5`.

***Вам потрібно налаштувати:***

1. **Синхронізацію стейт-файлів у S3** з використанням DynamoDB для блокування.
2. **Мережеву інфраструктуру (VPC)** з публічними та приватними підмережами.
3. **ECR (Elastic Container Registry)** для зберігання Docker-образів.

***Структура проєкту:***

```text
lesson-5/
│
├── main.tf                  # Головний файл для підключення модулів
├── backend.tf               # Налаштування бекенду для стейтів (S3 + DynamoDB)
├── outputs.tf               # Загальне виведення ресурсів
│
├── modules/                 # Каталог з усіма модулями
│   │
│   ├── s3-backend/          # Модуль для S3 та DynamoDB
│   │   ├── s3.tf            # Створення S3-бакета
│   │   ├── dynamodb.tf      # Створення DynamoDB
│   │   ├── variables.tf     # Змінні для S3
│   │   └── outputs.tf       # Виведення інформації про S3 та DynamoDB
│   │
│   ├── vpc/                 # Модуль для VPC
│   │   ├── vpc.tf           # Створення VPC, підмереж, Internet Gateway
│   │   ├── routes.tf        # Налаштування маршрутизації
│   │   ├── variables.tf     # Змінні для VPC
│   │   └── outputs.tf       # Виведення інформації про VPC
│   │
│   └── ecr/                 # Модуль для ECR
│       ├── ecr.tf           # Створення ECR репозиторію
│       ├── variables.tf     # Змінні для ECR
│       └── outputs.tf       # Виведення URL репозиторію ECR
│
└── README.md                # Документація проєкту
```

## **Кроки виконання завдання:**

1. **Створіть основну структуру проєкту**

   У кореневій папці `lesson-5` створіть файли:

   - `main.tf` — підключення модулів
   - `backend.tf` — налаштування бекенду для збереження стейтів у S3
   - `outputs.tf` — загальні вихідні дані з усіх модулів

2. **Налаштуйте S3 для стейтів і DynamoDB**

   У модулі `s3-backend`:

   - Налаштуйте **S3-бакет** для стейт-файлів Terraform
   - Увімкніть **версіювання** для збереження історії стейтів
   - Налаштуйте таблицю **DynamoDB** для блокування стейтів

   Виведення має відбуватись у `outputs.tf` URL S3-бакета та ім'я DynamoDB.

3. **Побудуйте мережеву інфраструктуру (VPC)**

   У модулі `vpc`:

   - Створіть **VPC** з CIDR блоком
   - Додайте **3 публічні підмережі** та **3 приватні підмережі**
   - Створіть **Internet Gateway** для публічних підмереж
   - Створіть **NAT Gateway** для приватних підмереж
   - Налаштуйте **маршрутизацію** через Route Tables

4. **Створіть репозиторій ECR**

   У модулі `ecr`:

   - Створіть ECR-репозиторій з автоматичним **скануванням образів**
   - Налаштуйте політику доступу для репозиторію
   - **Виведіть** URL репозиторію через `outputs.tf`

5. **Підключіть усі модулі в `main.tf`**

```js
# Підключаємо модуль S3 та DynamoDB
module "s3_backend" {
  source      = "./modules/s3-backend"
  bucket_name = "ваше ім'я"
  table_name  = "terraform-locks"
}

# Підключаємо модуль VPC
module "vpc" {
  source             = "./modules/vpc"
  vpc_cidr_block     = "10.0.0.0/16"
  public_subnets     = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  private_subnets    = ["10.0.4.0/24", "10.0.5.0/24", "10.0.6.0/24"]
  availability_zones = ["us-west-2a", "us-west-2b", "us-west-2c"]
  vpc_name           = "lesson-5-vpc"
}

# Підключаємо модуль ECR
module "ecr" {
  source      = "./modules/ecr"
  ecr_name    = "lesson-5-ecr"
  scan_on_push = true
}
```

6. **Налаштуйте бекенд для Terraform**

   Створіть `backend.tf` для налаштування S3 як бекенду:

```js
terraform {
  backend "s3" {
    bucket         = "ваше ім'я"
    key            = "lesson-5/terraform.tfstate"
    region         = "us-west-2"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}
```

7. **Зробіть документацію проєкту в `README.md`**

   У файлі `README.md` додайте:

   - Опис структури проєкту
   - Команди для ініціалізації та запуску:

   ```js
   terraform init
   terraform plan
   terraform apply
   terraform destroy
   ```

   - Пояснення кожного модуля: `s3-backend`, `vpc`, `ecr`

8. **Завантажте проєкт у репозиторій**

   - Створіть нову гілку `lesson-5`

   ```sh
   git checkout -b lesson-5
   ```

   - Додайте зміни в гілку

   ```sh
   git add .
   git commit -m "Add Terraform modules for S3, VPC, and ECR"
   git push origin lesson-5
   ```

   >!УВАГА! ⚠️ При роботі з хмарними провайдерами завжди пам'ятайте: невикористані ресурси можуть призвести до значних витрат. Щоб уникнути непередбачуваних рахунків, після перевірки вашого коду обов'язково видаляйте створені ресурси. Використовуйте команду terraform destroy.

   >!УВАГА! ⚠️ Пам'ятайте порядок запуску інфраструктури після видалення! При видаленні всієї інфраструктури за допомогою terraform destroy ви також видаляєте S3-бакет і DynamoDB-таблицю, які використовуються для збереження Terraform стейту.

# AWS Infrastructure with Terraform (Автоматизований підхід)

Цей проєкт розгортає базову інфраструктуру в AWS (VPC, ECR, S3 + DynamoDB) за допомогою Terraform модулів. 

У проєкті реалізовано підхід **"Автопілот"**. Вам не потрібно встановлювати Terraform, AWS CLI чи Python локально, а також не потрібно вручну запускати Docker-контейнери. Уся магія керується через кросплатформний `Makefile`.

Щоб побачити всі доступні команди, просто введіть у терміналі:
```bash
make help
```

## Структура проєкту

Проєкт використовує модульну архітектуру:

- `s3-backend`: Створює S3-бакет для збереження `.tfstate` та таблицю DynamoDB для State Locking.
- `vpc`: Створює ізольовану мережу (VPC) з 3 публічними та 3 приватними підмережами, IGW та NAT Gateway.
- `ecr`: Створює Elastic Container Registry для збереження Docker-образів з увімкненим скануванням вразливостей.

## 🚀 Як користуватися

### Опція А: Тестування локально (Безкоштовно через LocalStack)

*Ідеально для розробки та перевірки коду. Скрипт сам розбудить Docker, підніме LocalStack і розгорне інфраструктуру.*

> ⚠️ **Важливо:** Переконайтеся, що файл `backend.tf` повністю закоментований (щоб Terraform не шукав реальний AWS S3)!

```bash
# Розгортання середовища dev (за замовчуванням)
make deploy-local

# Або розгортання середовища prod локально
make deploy-local prod
```

### Опція Б: Деплой в реальний AWS (Бойовий запуск)

*Створення інфраструктури в хмарі відбувається у два етапи (вирішення проблеми "курки та яйця", оскільки S3 бакет для стейту ще не існує).*

**Етап 1: Створення базових ресурсів**
> ⚠️ Переконайтеся, що файл `backend.tf` закоментований!
```bash
make deploy-aws prod
```

**Етап 2: Міграція стейту в хмару S3**
1. Розкоментуйте вміст файлу `backend.tf`.
2. Виконайте команду розгортання ще раз (вона автоматично підхопить прапорець `-migrate-state`):
```bash
make deploy-aws prod
```
*(На питання Terraform у терміналі дайте відповідь `yes`)*

## 🧹 Видалення інфраструктури та очищення

> ⚠️ **УВАГА:** Щоб уникнути зайвих витрат в AWS, обов'язково видаляйте ресурси після роботи!

**Для AWS:** Оскільки S3 бакет стейту видаляється разом з інфраструктурою, нам потрібно повернути збереження стейту на локальний комп'ютер перед видаленням.
1. Закоментуйте вміст файлу `backend.tf`.
2. Виконайте команду для повернення стейту (на питання дайте відповідь `yes`):

   ```bash
   make deploy-aws prod
   ```

**Для LocalStack (локально):**

```bash
# Видалення ресурсів всередині емулятора
make destroy-local dev
```

**Глибоке очищення робочого місця:**

```bash
# Видалити кеш Terraform (.terraform), локальні стейти та базу LocalStack
make clean

# Повністю видалити всі Docker-образи проєкту та очистити пам'ять
make deep-clean
```

### 🌍 Середовища (Environments): `dev` vs `prod`

Наш код є адаптивним. Залежно від переданого середовища (аргумент у Makefile), Terraform автоматично змінює архітектуру без жодних ручних правок у коді.

🟢 **Середовище `dev` (Для локальної розробки)**

- **З чим працює:** Емулятор LocalStack (Community Edition) на вашому комп'ютері.
- **Що підтримує:** Базове тестування інфраструктури без ризику витратити реальні гроші.
- **Як адаптується:** Terraform автоматично оптимізує код під безкоштовний йомулятор:
   - **Вимикає** платні/непідтримувані функції (NAT Gateway, ECR Vulnerability Scanner).
   - **Вмикає** спеціальні налаштування маршрутизації (`s3_use_path_style`) для коректної роботи всередині Docker-мережі.

🔴 **Середовище `prod` (Бойовий або Pro-режим)**

- **З чим працює:** Справжня хмара **AWS** або локальний **LocalStack Pro** (за наявності ліцензії).
- **Що підтримує:** Повноцінний, відмовостійкий стек інфраструктури, готовий до високих навантажень.
- **Як адаптується:** Terraform розгортає архітектуру на максимум:
   - **Вмикає** NAT Gateway для доступу приватних підмереж до інтернету.
   - **Активує** сканування Docker-образів на вразливості при завантаженні в ECR (`scan_on_push = true`).

### 🔑 Безпека та ліцензії (Secret Management)

Для локального тестування `prod` конфігурації потрібен ліцензійний ключ LocalStack Pro. З міркувань безпеки файли `.tfvars` зі справжніми ключами **ніколи не комітяться в репозиторій**.

1. Скопіюйте шаблони `dev.tfvars.example` та створіть з них власні `dev.tfvars` / `prod.tfvars` (вони ігноруються Git).
2. Якщо у вас є ключ LocalStack Pro, створіть файл `.env` на базі `.env.example`.
3. Вставте ваш ключ: `LOCALSTACK_AUTH_TOKEN=ls-...`
4. При наступному запуску інфраструктури, Docker автоматично підхопить ключ і активує Pro-функціонал.
